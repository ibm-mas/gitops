## Overview

The AI Service Tenant Root Application orchestrates the deployment of a single AI Service tenant by rendering the tenant ArgoCD Application. This application is generated by the AI Service Instance Root's Tenant ApplicationSet.

### Deployment Sequence

Applications are deployed in the following order (controlled by sync-wave annotations):

1. **100: AI Service Tenant** - Tenant operator subscription and tenant custom resource

## Configuration

**Important:** This application receives its configuration from the parent AI Service Instance Root ApplicationSet. Values are not read from a `values.yaml` file but are passed as Helm parameters from the ApplicationSet generator.

## Configuration Files

Configuration comes from files in the config repository that match the patterns defined in the AI Service Instance Root's Tenant ApplicationSet:

| File Pattern | Required | Purpose |
|--------------|----------|---------|
| `{account.id}/{cluster.id}/{aiservice_instance_id}/{tenant_id}/aiservice-tenant-params.yaml` | Yes | Tenant-specific configuration (tenant ID, namespace, licensing, WatsonX, scheduling) |

See the [AI Service Instance Root README](../ibm-aiservice-instance-root/README.md) for more information about the parent ApplicationSet.

### Role Conditions

- **Cluster Admin Role**: Applications that require `cluster_admin_role` to be set (0 applications)
- **Application Admin Role**: Applications that require `application_admin_role` to be set (0 applications)
- **Both Roles**: Applications rendered regardless of role settings (1 application)

**Note**: The tenant application has additional conditions beyond role requirements (e.g., specific configuration values must be defined). Refer to the template file for complete rendering logic.
